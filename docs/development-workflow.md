# 开发工作流

## 基本原则

- 先实现本地闭环，再扩展在线能力。
- 每个会改变世界状态的功能都必须考虑存档和回滚。
- 每个 AI 调用都必须有 fixture，不能只靠真实服务测试。
- 文档中的数据结构变更后，需要同步更新示例和迁移策略。

## 推荐实现顺序

1. 搭建目录和基础场景。
2. 实现 `WorldStore` 和存档。
3. 实现世界创建表单。
4. 接入 AI 请求抽象，但先用 mock 响应。
5. 接入真实 Glosc One。
6. 实现事件历史和记忆检索。
7. 加入地图和 NPC 模拟。

## 目录约定

Tauri/Vue 资源：

```text
src/views
src/stores
src/domain
src-tauri/src
public/assets
```

文档：

```text
docs/
```

测试资料：

```text
tests/fixtures/ai/
tests/fixtures/saves/
```

## 提交前检查

每次提交前至少确认：

- 项目能启动。
- 新增脚本无明显语法错误。
- 新增数据结构有文档说明。
- 修改 AI 响应格式时同步 fixture。
- 修改存档结构时增加 schema version 或迁移说明。
- 涉及 UI 的改动至少检查桌面、手机、平板三个尺寸。

## 文档维护

需要更新文档的情况：

- 新增核心玩法。
- 新增或修改数据模型字段。
- 新增 AI 请求类型。
- 修改存档格式。
- 修改 Glosc One 调用方式。
- 修改平台适配、输入方式或响应式布局。
- 修改 MVP 范围。

## 调试建议

- AI 原始请求和响应写入独立日志。
- 世界状态 patch 在应用前后都可查看。
- 存档加载失败时输出具体文件和字段。
- UI 中提供开发模式开关，但发布版本默认关闭。

## 版本命名

建议：

- `0.1.x`：项目骨架和创建流程。
- `0.2.x`：探索循环。
- `0.3.x`：记忆系统。
- `0.4.x`：地图和 NPC 模拟。
- `0.5.x`：公开测试前打磨。
