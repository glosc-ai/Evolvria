# 游戏设计规格

## 核心循环

1. 玩家创建世界、主角和关键角色。
2. 系统生成结构化世界和开局事件；若已配置 Glosc One，会记录一次世界扩写调用。
3. 玩家选择推荐行动或自由输入行动。
4. AI 或本地 mock 返回叙事结果和结构化更新。
5. 系统写入时间线、记忆、关系、地点状态和线索进度。
6. 玩家查看地图、人物、地点、时间线、线索和世界观。
7. 玩家继续探索，或运行 NPC tick 让非玩家事件推进。

## 玩家创建内容

当前 `WorldSeed` 包含：

- 世界名称。
- 类型：奇幻、科幻、现代都市、武侠等。
- 基调：冒险、政治、悬疑、温情等。
- 内容偏好与禁用内容。
- 叙事详细度。
- NPC 自主频率。
- 主角：姓名、身份描述、目标、能力、弱点。
- 关键角色：姓名、身份、与主角关系、性格、目标、秘密、行动倾向、描述。

关键角色不应被性别模板锁死；默认样例只是加速创建。

## 探索方式

当前采用文字输入加推荐行动：

- 玩家可自由输入行动。
- 推荐行动来自 `suggested_actions`。
- 结果会生成 timeline event。
- 重要叙事变化应通过 patch、memory、relationship 或 thread progress 进入状态。

如果玩家行动不合理，AI 应给出受阻、部分成功或代价成功，而不是无条件成功。

## 时间系统

当前 `PlayerActionResult.time_delta_minutes` 会转换为小时推进，更新事件时间和世界当前时间。

默认粒度：

- 即时事件：对话、检查、短动作。
- 短时推进：几十分钟到几小时。
- 长时推进：半天、一天、多天。

每个事件记录地点、参与角色、原因、结果和影响。

## 角色设计

角色包含：

- 叙事字段：描述、性格、目标、秘密、关系。
- 模拟字段：当前位置、状态、行动倾向、是否同行、可见性。
- 记忆字段：`memory_summary`、`known_event_ids`、相关 `Memory`。
- 玩家备注：`player_notes` 和更新时间。

NPC 自主行动当前是简化 tick：选择一个 active NPC，写入 `npc_simulation` 事件和 AI 日志。后续可接入真实 AI 事件生成和移动规则。

## 地点与地图

地点包含坐标、可见性、状态标签、事件列表、玩家备注、biome 和 height。地图当前以 SVG 展示地点和路线；图片底图是后续 UI 接入项。

地图设计原则：

- 位置坐标不可被 AI 覆盖玩家确认值。
- 未知地点默认隐藏，可手动显示。
- 移动到地点会让地点变为已知。
- 同行角色随玩家移动。

## 事件类型

常用类型：

- `world_event`
- `player_action`
- `npc_simulation`
- `relationship_change`
- `location_change`
- `discovery`
- `conflict`
- `quest`

类型是字符串，允许扩展，但 UI 筛选和测试应覆盖新增核心类型。

## 线索线程

线程用于追踪未解决问题、主线、关系线和任务线。

- `status = open | resolved`
- `priority` 决定重要度。
- `progress` 记录事件推进。

玩家行动成功后，当前实现会推进最多两个 open thread。

## 失败与不确定性

行动结果建议包括：

- 确定成功。
- 代价成功。
- 部分成功。
- 失败但产生新局面。

结果应写入：

- `outcome`
- `outcome_reason`
- `consequence`

失败不等于无变化；失败也可以生成线索、关系变化或地点状态变化。
